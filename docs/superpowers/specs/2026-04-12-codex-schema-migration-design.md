# Codex Generated Schema Migration

## Context

The Hive Codex integration hand-writes all JSON-RPC protocol types (`JsonRpcRequest`, `JsonRpcResponse`, notification params, approval decisions, etc.) rather than using the schemas generated by `codex app-server generate-ts`. This has led to protocol mismatches that cause tools not to show correctly, agents not to be waited on properly, and sessions to end prematurely compared to the official Codex app.

The Codex CLI (v0.118.0) can generate canonical TypeScript types for the entire app-server protocol. This migration replaces all hand-written protocol types with the generated ones, fixing known mismatches and preventing future drift.

## Protocol Mismatches Found

These are confirmed bugs where our hand-written types diverge from the generated schema:

| # | Mismatch | Our Code | Generated Schema | Impact |
|---|----------|----------|------------------|--------|
| 1 | **Approval decisions** | `'once' \| 'always' \| 'reject'` | `'accept' \| 'acceptForSession' \| 'decline' \| 'cancel'` | Codex likely ignores/errors on our approval responses |
| 2 | **ThreadStartParams** | Missing `experimentalRawEvents`, `persistExtendedHistory` | Both required `boolean` fields | Codex may reject or use unexpected defaults |
| 3 | **ThreadResumeParams** | Missing `persistExtendedHistory` | Required `boolean` | Same as above |
| 4 | **TurnStartParams.effort** | `settings: { reasoningEffort }` in collaborationMode | Top-level `effort?: ReasoningEffort` field | Reasoning effort may not be applied outside collaborationMode |
| 5 | **UserInput text_elements** | Not sent | Required `text_elements: Array<TextElement>` | Codex may reject text input |
| 6 | **FileChange approval** | Uses command execution decision type | Separate `FileChangeApprovalDecision` type | File approvals may not be processed correctly |

### Phase 1: Schema Foundation

**Goal:** Copy generated schemas into the project and verify they compile alongside existing code.

**Tasks:**
1. Copy `/tmp/codex-schemas/` contents → `src/shared/codex-schemas/` (including `v2/` and `serde_json/` subdirectories)

### Phase 2: Protocol Layer Migration — `codex-app-server-manager.ts`

**Goal:** Replace all hand-written protocol types in the manager with generated schema imports. Fix all known protocol mismatches. This is the critical phase — the manager is the single chokepoint for all Codex I/O.

**Tasks:**

#### 2a. Replace JSON-RPC envelope types
Remove hand-written `JsonRpcRequest`, `JsonRpcResponse`, `JsonRpcNotification`, `JsonRpcError` interfaces. Import `ClientRequest`, `ClientNotification`, `ServerRequest`, `ServerNotification` from generated schemas. The JSON-RPC envelope itself (jsonrpc, id fields) remains as a thin wrapper since the generated types model the application-level protocol, not the transport envelope.

#### 2b. Type request methods with generated params
Replace `CodexStartSessionOptions` and `CodexTurnInput` with types derived from generated `ThreadStartParams`, `ThreadResumeParams`, `TurnStartParams`. Map:
- `startSession()` → uses `ThreadStartParams` (add `experimentalRawEvents: false`, `persistExtendedHistory: false`)
- `sendTurn()` → uses `TurnStartParams` (fix `effort` field name, add `text_elements: []` to UserInput)
- `readThread()` → uses `ThreadReadParams`
- `rollbackThread()` → uses `ThreadRollbackParams`
- `interruptTurn()` → uses `TurnInterruptParams`

#### 2c. Type response handling with generated response types
Replace untyped `asObject(response)` narrowing with generated `ThreadStartResponse`, `ThreadResumeResponse`, `TurnStartResponse`. Extract `thread.id` directly instead of probing with `asString(threadObj?.id)`.

#### 2d. Fix approval decision values
Replace the `'once' | 'always' | 'reject'` decision type with a mapping function:
```typescript
function toCodexApprovalDecision(
  hiveDecision: 'once' | 'always' | 'reject',
  method: string
): CommandExecutionApprovalDecision | FileChangeApprovalDecision {
  // 'once' → 'accept', 'always' → 'acceptForSession', 'reject' → 'decline'
}
```
The Hive-internal values (`once`/`always`/`reject`) remain in the UI layer; the manager maps them to Codex values at the protocol boundary.

#### 2e. Type server notifications and requests
Replace the generic `isServerRequest`/`isServerNotification`/`isResponse` type guards with guards that narrow to the generated `ServerNotification` and `ServerRequest` discriminated unions. The `method` field becomes the discriminant for typed payload access.

#### 2f. Type the `handleServerNotification` handler
Replace `asObject(notification.params)` narrowing with direct typed access:
```typescript
// Before
const turnObj = asObject(asObject(notification.params)?.turn)
const turnId = asString(turnObj?.id)

// After (with TurnStartedNotification type)
const turnId = notification.params.turn.id
```

#### 2g. Type the `handleServerRequest` handler  
Replace generic payload access with typed `CommandExecutionRequestApprovalParams`, `FileChangeRequestApprovalParams`, `ToolRequestUserInputParams`.

**Files modified:**
- `src/main/services/codex-app-server-manager.ts` — major refactor of types, method signatures, and narrowing logic

**Definition of Done:**
- [ ] No hand-written JSON-RPC protocol types remain in the manager (except the thin JSON-RPC 2.0 envelope wrapper)
- [ ] `ThreadStartParams` includes `experimentalRawEvents: false` and `persistExtendedHistory: false`
- [ ] `ThreadResumeParams` includes `persistExtendedHistory: false`
- [ ] Approval responses send `'accept'`/`'acceptForSession'`/`'decline'` instead of `'once'`/`'always'`/`'reject'`
- [ ] `UserInput` text entries include `text_elements: []`
- [ ] `tsc --noEmit` passes
- [ ] The `codex.jsonl` debug log (from the earlier logging PR) shows correct field names and values in outgoing requests

**Testing:**
1. Start a Codex session → verify `codex.jsonl` shows `thread/start` with `experimentalRawEvents` and `persistExtendedHistory` fields
2. Send a prompt → verify `turn/start` params use correct `effort` field and `UserInput` format with `text_elements`
3. Trigger a tool execution that requires approval → verify `codex.jsonl` shows `'accept'` (not `'once'`) in the approval response

### Phase 3: Event Mapper Migration — `codex-event-mapper.ts`

**Goal:** Replace `unknown` payload types in the event mapper with generated notification types so payload extraction is type-safe.

**Tasks:**

#### 3a. Type notification payload extraction
Replace `asObject(event.payload)` + `asString(...)` chains with typed params. The `CodexManagerEvent.payload` can carry the generated notification params type, narrowed by `event.method`.

For each extract function:
- `extractContentDelta()` → type with `AgentMessageDeltaNotification`, `ReasoningTextDeltaNotification`, `CommandExecutionOutputDeltaNotification`, `FileChangeOutputDeltaNotification`, `PlanDeltaNotification`
- `extractTurnCompletedInfo()` → type with `TurnCompletedNotification` (which contains `Turn` with typed `TurnStatus`)
- `extractItemInfo()` → type with `ItemStartedNotification` / `ItemCompletedNotification` (which contain `ThreadItem` discriminated union)
- `extractTaskInfo()` → type with relevant task notification params

#### 3b. Use ThreadItem discriminated union for tool classification
Replace string-based tool type matching with the `ThreadItem` discriminated union (`type: 'commandExecution' | 'fileChange' | 'mcpToolCall' | ...`). The `isToolLifecycleItem()` function can use the `type` discriminant directly.

#### 3c. Refine CodexManagerEvent payload type
Update the `CodexManagerEvent` interface to optionally carry a typed payload:
```typescript
export interface CodexManagerEvent {
  // ...existing fields...
  payload?: ServerNotification['params'] | ServerRequest['params']
}
```
This gives downstream consumers (event mapper, activity mapper) type information when they narrow by `method`.

**Files modified:**
- `src/main/services/codex-event-mapper.ts`
- `src/main/services/codex-app-server-manager.ts` (CodexManagerEvent type refinement)

**Definition of Done:**
- [ ] All `asObject(event.payload)` chains in `codex-event-mapper.ts` are replaced with typed access
- [ ] `ThreadItem` type discriminant used for tool classification
- [ ] `tsc --noEmit` passes
- [ ] No `as unknown` or `as any` casts added (type-safe narrowing only)

**Testing:**
1. Start a Codex session and send a coding task → verify streaming text appears correctly
2. Verify tool cards render (command execution, file changes)
3. Verify thinking/reasoning blocks render
4. Verify token usage updates display
5. Verify session title generation works
6. Compare `codex.jsonl` output against a trace from the official Codex app for the same prompt

### Phase 4: Implementer Migration — `codex-implementer.ts`

**Goal:** Use typed params/responses in the implementer layer. Fix HITL (approval/question) flows to use generated types.

**Tasks:**

#### 4a. Type turn construction
Replace manual `params: Record<string, unknown>` construction in `prompt()` with `TurnStartParams`. Use `UserInput` type for input array construction with required `text_elements` field.

#### 4b. Type approval handling
The implementer's `permissionReply()` method currently passes `'once'|'always'|'reject'` to the manager. After Phase 2, the manager already handles the mapping, so the implementer just needs updated type annotations. Also type the permission request construction using `CommandExecutionRequestApprovalParams` fields.

#### 4c. Type user input handling
Use `ToolRequestUserInputParams` for question extraction and `ToolRequestUserInputResponse` for answer construction. Replace `CodexUserInputAnswer` hand-written type with `ToolRequestUserInputAnswer` from generated schemas.

#### 4d. Type thread snapshot parsing
When parsing `thread/read` responses, use `Thread` type with `Turn[]` and `ThreadItem` discriminated union for structured extraction instead of `asObject` chains.

#### 4e. Type session title extraction
Use `ThreadNameUpdatedNotification` type for title event handling.

**Files modified:**
- `src/main/services/codex-implementer.ts`

**Definition of Done:**
- [ ] No `asObject`/`asString` narrowing used for protocol payloads (only for truly unknown data)
- [ ] `CodexUserInputAnswer` replaced with `ToolRequestUserInputAnswer` import
- [ ] Turn input construction uses `UserInput` and `TurnStartParams` types
- [ ] `tsc --noEmit` passes

**Testing:**
1. Full session lifecycle: connect → prompt → receive response → disconnect
2. Approval flow: trigger a command needing approval → approve → verify execution continues
3. User input flow: trigger `request_user_input` (plan mode) → answer → verify agent continues
4. Undo/rollback: verify `thread/rollback` works with typed params
5. Reconnect: verify `thread/resume` works with typed params including `persistExtendedHistory`
6. Verify the session runs for the same duration as the official Codex app (the original symptom)
